# v0.3.0

Release etiketi: `v0.3.0`

## Sürümün Odağı

CacheDB `0.3.0`; Redis'teki eski verinin yeni veriyi ezmesi, geniş sorguların
JVM belleğini sınırsız kullanması, başarısız SQL yazılarının başarılı kabul
edilmesi ve çok pod'lu tenant kota yarışları gibi production risklerini
sertleştirir. Önceki public constructor ve metot imzaları, geriye uyumlu
delegating overload'larla korunur.

## Eklenenler

- Warm, read-through ve dış değişiklik akışları için sürüm kontrollü JDBC
  kaynak okuma ve Redis'te atomik compare-and-set hydration.
- Açık stale-write ve Redis backpressure hata kategorileri.
- JDBC okuma ve yazma işlemleri için SQL zaman aşımı.
- Sınırlı admin istek/arka plan kuyrukları, request body limiti ve Redis'te
  tutulan job durumu.
- Indexed ve degraded sorgu yolları için aday kayıt sınırı.
- Redis Function kaynağı için SHA-256 özeti, semantik dağıtım sürümü ve lease
  korumalı yükleme akışı.
- Eski hydration, tenant kota yarışı, geniş sorgu, function drift ve admin body
  limiti için Docker integration testleri.

## Değişenler

- Kalıcı compaction payload, pending state ve version fence kayıtları artık
  expire edilmez. Kalıcı compaction stream'i acknowledgement öncesinde
  trim edilemez.
- Prefix ve token index'leri varsayılan olarak kapalıdır.
- Geniş sorgular büyük JVM listeleri oluşturmadan önce reddedilir.
- Index'te bulunup payload'ı eksik olan kayıtlar sınırlı loader ile onarılır
  veya sorgu açık hata verir; eksik sonuç sayfası sessizce dönmez.
- Tenant satır ve payload kabul kararı, pod'lar arasında tenant bazında
  sıralanır.
- Repository graph'ları ve CacheDB'nin açtığı Redis client'ları tekrar kullanılır
  ve kontrollü biçimde kapatılır.
- Redis Function yükleme işlemi aynı sürümde kaynak değişikliğini ve sürüm
  düşürmeyi reddeder.

## Düzeltilenler

- Eski warm, read-through veya CDC sonucu Redis'teki daha yeni kaydı ezemez ve
  yeni tombstone'u yeniden canlandıramaz.
- Sıfır satır etkileyen SQL yazısı, kalıcı sürüm kontrol edilmeden başarılı
  kabul edilmez.
- Eş zamanlı kota kararı sırasında çıkarılan tenant kaydı sonradan yeniden
  eklenemez.
- Reddedilen admin işleri süresiz biçimde `QUEUED` durumunda kalmaz.
- Production test runner'ları tek ve açık ortam çözümleme sözleşmesi kullanır.

## v0.2.0'dan Geçiş Notları

1. Tüm CacheDB modüllerini ve annotation processor'ı birlikte `0.3.0` yap.
2. `redisFunctions.libraryVersion` değerini dağıtılan function kaynağıyla aynı
   sürümde tut. Lua kaynağını değiştiriyorsan dağıtım sürümünü de değiştir.
3. Kalıcı compaction payload, pending ve version TTL değerlerini `0` bırak.
   Sıfırdan büyük değerler replay güvenliğini bozabileceği için reddedilir.
4. Prefix/token index kullanan yolları gözden geçir. Bu index'ler artık
   varsayılan olarak kapalıdır ve yalnızca sınırlı, ölçülmüş yollar için
   açılmalıdır.
5. Geniş entity sorgularını kontrol et. `maxMaterializedCandidateIds` sınırını
   aşan aday listeleri artık sınırsız JVM allocation yapmak yerine açık hata
   verir.
6. Redis'i doldurması gereken özel source loader'lar pozitif entity sürümünü
   versioned loader sözleşmesiyle sağlamalıdır.
7. Read-through ve write-behind SQL zaman aşımını provider davranışına ve route
   SLO'suna göre belirle; veritabanı çağrılarını sınırsız bırakma.

## Önerilen Dependency

```xml
<properties>
    <cachedb.version>0.3.0</cachedb.version>
</properties>

<dependency>
    <groupId>com.reactor.cachedb</groupId>
    <artifactId>cachedb-spring-boot-starter</artifactId>
    <version>${cachedb.version}</version>
</dependency>
```

MSSQL uygulaması aynı sürümde `cachedb-storage-mssql` dependency'sini de
eklemelidir.

doğrulamalıdır.
