# v0.5.0

Release etiketi: `v0.5.0`

CacheDB `0.5.0`, birden fazla container veya Kubernetes pod'u ile çalışan
uygulamalar için deklaratif periyodik warm ve artımlı aktif veri reconciliation
desteği getirir.

## Öne Çıkanlar

- `@CacheScheduledWarm`; cron, sabit gecikme ve sabit hız planlarını Spring property değerleriyle kullanabilir.
- Redis lease, pod'lar arasından tek bir warm sahibi seçer; heartbeat uzun süren işlerde lease süresini yeniler.
- Lease alamayan pod'lar sınırlı bir süre bekler, ardından scheduler thread'ini süresiz tutmadan o çevrimi atlar.
- Başarılı çevrim kaydı ve `minimumIntervalString`, farklı pod'ların aynı warm işini art arda tekrar etmesini önler.
- Warm sahibi hata verirse tamamlanma kaydı yazılmaz; başka bir pod işi güvenli biçimde yeniden deneyebilir.
- Cursor tabanlı reconciliation, aktif veri policy'sine artık uymayan entity, projection ve indeks verilerini kaldırır.
- Eksik veya bozuk Redis payload'ları yalnızca cache'ten çıkarılır ve cursor ilerlemesini durdurmaz.
- Çalışma durumu; yüklenen, gönderilen, incelenen, silinen, eksik, bozuk, cursor ve tam çevrim değerlerini gösterir.
- PostgreSQL ve MSSQL örneklerinde doğrudan kullanılabilen periyodik sipariş planı ve operasyon endpoint'i bulunur.

## Kullanılacak Sürüm

```xml
<properties>
    <cachedb.version>0.5.0</cachedb.version>
</properties>
```

## Doğrulama

- 13 modüllü ana reactor: 250 test, 0 hata, 0 başarısız test ve ortama bağlı 12 atlanan test.
- Çok podlu lease, heartbeat, warm sahibinin hata vermesi ve başka pod'un işi devralması canlı Redis üzerinde doğrulandı.
- Cache-only reconciliation; bozuk payload temizliği ve write-behind stream'inin değişmediği kanıtıyla canlı Redis ve PostgreSQL üzerinde geçti.
- PostgreSQL örneği 7 birim testi ve canlı Redis/PostgreSQL provider integration testini geçti.
- MSSQL örneği 7 birim testi ve canlı Redis/SQL Server provider integration testini geçti.
- Public API compatibility, Türkçe dokümantasyon, Postman JSON ve whitespace kontrolleri geçti.

Standart reactor'da Redis bağlantısı beklediği için atlanan iki coordination
testi, ayrı canlı Redis koşusunda da başarıyla tamamlandı. Kalan 10 test, özel
SQL Server profili arkasındaki canlı SQL Server testleridir. MSSQL örneğinin
provider integration testi gerçek bir SQL Server 2022 container'ı üzerinde
çalıştırıldı.

## Production Sınırı

Periyodik warm, kaynak verisi için sınırlı gecikme sağlar; sıfır gecikmeli CDC
yerine geçmez. Doğrudan SQL'e yazılan bir kayıt, outbox/CDC CacheDB'yi ayrıca
beslemiyorsa bir sonraki başarılı warm çevriminden sonra Redis'e gelir.
`CacheWarmPlan.maxRows`, JDBC sınırları, Redis bellek bütçesi ve aktif veri
policy'si hedeflenen route penceresini kapsamalıdır. Doğrudan SQL değişiklikleri
Redis'e çok kısa sürede gelmek zorundaysa outbox/CDC kullanılmalıdır.

Redis lease normal koşullarda eş zamanlı çalışmayı önler, ancak distributed
transaction değildir. Uzun JVM duraklaması veya Redis bağlantı kopması
at-least-once çakışma oluşturabilir. Warm hydration sürüm kontrollü ve
idempotent çalışır; tamamlanma kaydı yalnızca pod lease sahibi olmaya devam
ederken yazılır.
