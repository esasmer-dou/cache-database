# v0.3.1

Release etiketi: `v0.3.1`

## Bu Patch Neden Çıktı?

CacheDB `0.3.0`, kalıcı SQL sonucunu doğrulayan yeni kontroller ekledi. Ancak
ilk MSSQL evidence koşusunda SQL Server'da daha yeni bir sürüm bulunmasına
rağmen kuyruktan sonra gelen eski yazı hata kabul edildi. Write-behind
sisteminde bu durum zaten kalıcı olarak karşılanmıştır. Eski komutu tekrar
denemek veya dead-letter'a taşımak doğruluğu artırmaz; yalnızca ek yük üretir.

`0.3.1`, version fence'i zayıflatmadan bu ayrımı düzeltir.

## Düzeltilenler

- UPSERT için `currentVersion >= incomingVersion`, idempotent tekrar veya eski
  komut no-op'u olarak kabul edilir.
- DELETE için satırın bulunmaması veya `currentVersion > incomingVersion`,
  kalıcı no-op olarak kabul edilir.
- Veritabanındaki sürüm gelen komuttan eskiyse veya istenen silme sonucu
  oluşmamışsa sıfır-row sonucu hata vermeye devam eder.
- MSSQL'de aynı id için eş zamanlı yazılar ve yüksek hacimli eski sürüm
  batch'leri gereksiz retry/dead-letter baskısı oluşturmaz.
- PostgreSQL prepared, batch, multi-row ve COPY yolları ortak JDBC desteği
  üzerinden aynı kalıcı sonuç kuralını kullanır.

## Geçiş Rehberi

- `0.3.0` kullanan uygulamalar bütün CacheDB modüllerini ve annotation
  processor'ı birlikte `0.3.1` sürümüne yükseltmelidir.
- Entity, projection, şema veya configuration migration gerekmez.
- Bu patch'te Redis Function kaynağı değişmedi. Function'ın bağımsız varsayılan
  dağıtım sürümünün `0.3.0` kalması bilinçli bir karardır.

## Dependency

```xml
<properties>
    <cachedb.version>0.3.1</cachedb.version>
</properties>
```

## Doğrulama Kanıtı

- `0.3.0` hardening reaktöründe `222` test, `0` failure, `0` error ve ortam
  koşuluna bağlı `10` skip oluştu.
- `0.3.1` için odaklı MSSQL birim testleri geçti.
- Redis kesinti/kurtarma senaryolu yerel Docker HA preflight geçti.
- SQL Server provider evidence, container restart öncesi ve sonrasında geçti.
  Bu koşu duplicate insert yarışını, yüksek hacimli sürümlü yükü, outbox,
  migration discovery ve çok pod'lu apply-runner senaryolarını kapsadı.
- PostgreSQL ve MSSQL sample consumer workflow'ları, temiz Java 21 runner'larında
  yayımlanmış CacheDB paketini GitHub Packages'tan çekerek build oldu.

CacheDB'yi kullanan her uygulama; kendi veri eşleşmesini, Redis bellek bütçesini,
provider gecikmesini, rollback planını ve hedef production HA topolojisini ayrıca
doğrulamalıdır.
