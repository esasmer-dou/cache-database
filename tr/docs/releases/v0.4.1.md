# v0.4.1

Release etiketi: `v0.4.1`

CacheDB `0.4.1`, deklaratif 0.4 sürüm ailesinin production için yayımlanan
kararlı sürümüdür. 0.4.0 ile gelen generated domain scope, YAML tabanlı entity
politikaları, tip güvenli projection şemaları ve warm planlarına ek olarak,
release kontrolünün yakaladığı public API uyumluluk sorununu düzeltir.

## Öne Çıkanlar

- Tek tip güvenli domain API'si olarak paket düzeyinde `GeneratedCacheModule.Scope`.
- Kullanılmayan repository'leri oluşturmayan thread-safe ve lazy entity scope'ları.
- Composite policy ve hatalı entity adında fail-fast davranışı olan YAML tabanlı `CachePolicyCatalog`.
- Entity bazlı policy değerlerini koruyan iki aşamalı JDBC source ve relation-loader registration.
- Projection serialization ve index kolonları için tek sözleşme olan `ProjectionSchema`.
- Tip güvenli dry-run, yalnız projection ve tam warm çalıştırma seçenekleri.
- Başlangıçta Redis'in tamamını taramak yerine sınırlı ve kademeli planner telemetry taraması.
- Manuel repository veya policy factory içermeyen PostgreSQL ve MSSQL örnekleri.
- Geriye uyumlu `metadata-only` starter varsayılanı; JDBC yükleme açıkça seçilir.
- 0.4.0 öncesindeki Spring auto-configuration factory imzası, yönlendiren bir uyumluluk overload'u olarak korunur.

## Kullanılacak Sürüm

```xml
<properties>
    <cachedb.version>0.4.1</cachedb.version>
</properties>
```

## Doğrulama

- Java 21 üzerinde 13 modüllü reactor başarıyla tamamlandı.
- İzole Redis 8 ve PostgreSQL ortamında 83 integration testi geçti.
- Benchmark, recovery, coordination, fault injection ve certification kapsamındaki 27 production testinin tamamı geçti.
- İki REST örneği de unit testlerini ve gerçek Redis 8 ile seçilen SQL provider entegrasyonunu geçti.
- Public API karşılaştırmasında 66 ek imza satırı bulunuyor; önceki imzalardan hiçbiri kaldırılmadı.

## Production Sınırı

CacheDB hangi route'ların Redis'te tutulması gerektiğini kendiliğinden belirlemez.
Her uygulama cutover öncesinde sınırlı route sözleşmesini, aktif veri politikasını,
warm kapsamını, bellek bütçesini, SQL dönüş yolunu ve rollback kanıtını hazırlamalıdır.
