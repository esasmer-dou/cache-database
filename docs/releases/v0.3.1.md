# v0.3.1

Release tag: `v0.3.1`

## Why This Patch Exists

CacheDB `0.3.0` added durable SQL outcome verification, but its first MSSQL
evidence run treated an out-of-order older write as a failure when SQL Server
already contained a newer version. In a write-behind system that state is
already durably satisfied: retrying or dead-lettering the older command adds
load without improving correctness.

Version `0.3.1` corrects that distinction without weakening the version fence.

## Fixed

- UPSERT: `currentVersion >= incomingVersion` is an idempotent or stale durable
  no-op.
- DELETE: a missing row or `currentVersion > incomingVersion` is a durable
  no-op.
- A zero-row outcome still fails when the database contains an older version
  than the incoming command or does not satisfy the requested delete.
- MSSQL concurrent same-id writes and stale high-volume batches no longer
  produce avoidable retry/dead-letter pressure.
- PostgreSQL prepared, batch, multi-row, and COPY paths use the same durable
  outcome rule through shared JDBC support.

## Upgrade Guidance

- Applications using `0.3.0` should upgrade all CacheDB modules and the
  annotation processor to `0.3.1` together.
- No entity, projection, schema, or configuration migration is required.
- Redis Function source did not change in this patch. Its independent default
  deployment version remains `0.3.0`; this is intentional.

## Dependency

```xml
<properties>
    <cachedb.version>0.3.1</cachedb.version>
</properties>
```

## Validation Evidence

- Full `0.3.0` hardening reactor: `222` tests, `0` failures, `0` errors, `10`
  environment-conditional skips.
- Focused `0.3.1` MSSQL unit tests passed.
- Local Docker HA preflight passed with Redis outage/recovery.
- Live SQL Server provider evidence passed before and after container restart,
  including duplicate insert races, high-volume versioned load, outbox,
  migration discovery, and multi-pod apply-runner coverage.
- PostgreSQL and MSSQL sample consumer workflows resolve the published CacheDB
  package from GitHub Packages on clean Java 21 runners.

Each application must still validate its own data parity, memory budget,
provider latency, rollback plan, and target production HA topology.
