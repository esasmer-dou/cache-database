# v0.5.0

Release tag: `v0.5.0`

CacheDB `0.5.0` adds declarative periodic warm execution and incremental
active-data reconciliation for applications running multiple containers or
Kubernetes pods.

## Highlights

- `@CacheScheduledWarm` supports cron, fixed-delay, and fixed-rate plans with Spring property placeholders.
- A Redis lease selects one warm owner across pods; heartbeat renewal protects long-running jobs.
- Losing pods wait for a bounded interval, then skip instead of consuming scheduler threads indefinitely.
- Successful-cycle markers and `minimumIntervalString` prevent sequential duplicate warm work across instances.
- Failed owners do not write a completion marker, so another pod can safely retry.
- Cursor-based reconciliation removes entity, projection, and index state that no longer matches the active-data policy.
- Missing and invalid Redis payloads are removed cache-only and cannot poison cursor progress.
- Runtime snapshots expose loaded, submitted, inspected, evicted, missing, invalid, cursor, and full-cycle values.
- PostgreSQL and MSSQL samples include a copy-paste periodic order-window plan and operational status endpoint.

## Production Boundary

Periodic warm provides bounded source-data lag, not zero-lag CDC. A row written
directly to SQL appears after the next successful warm unless an outbox/CDC
adapter also feeds CacheDB. `CacheWarmPlan.maxRows`, JDBC guardrails, Redis
memory budgets, and the active-data policy must cover the intended route window.
Use outbox/CDC when direct SQL changes must reach Redis with near-zero delay.

The Redis lease prevents normal concurrent execution. It is not a distributed
transaction: an extended JVM pause or Redis partition can still create
at-least-once overlap. Warm hydration remains version-fenced and idempotent, and
the completion marker is committed only while the owner still holds leadership.
