# v0.2.0

Release tag: `v0.2.0`

## What Changed

- Upgraded the framework and sample projects to version `0.2.0`.
- Added Java 21 friendly annotation processor support by using the latest supported source version at compile time.
- Hardened the JDBC source loader for SQL Server by avoiding duplicate stable id columns in generated `ORDER BY` clauses.
- Hardened JDBC write-behind binding so nullable SQL columns, such as soft-delete markers, are preserved instead of failing with `NullPointerException`.
- Updated PostgreSQL and MSSQL samples to register generated JDBC-backed bindings with `registerJdbcBacked(...)`.
- Updated sample warm/backfill flows to use `CacheWarmPlan` and the registered `EntityQueryLoader` instead of ad-hoc `JdbcTemplate` warm logic.
- Added explicit read-through tuning in the samples with `ReadThroughConfig`.
- Added deterministic sample load-test scripts for hot timeline routes.
- Updated sample schemas with CacheDB runtime columns required by write-behind.
- Updated English and Turkish sample documentation with copy-paste dependency, warm, read-through, and load-test examples.

backlog checks, and JVM GC review.

## Recommended Dependency

```xml
<properties>
    <cachedb.version>0.2.0</cachedb.version>
</properties>

<dependency>
    <groupId>com.reactor.cachedb</groupId>
    <artifactId>cachedb-spring-boot-starter</artifactId>
    <version>${cachedb.version}</version>
</dependency>
```

```xml
<dependency>
    <groupId>com.reactor.cachedb</groupId>
    <artifactId>cachedb-storage-mssql</artifactId>
    <version>${cachedb.version}</version>
</dependency>
```

## Production Positioning

CacheDB `0.2.0` should be used as a Redis-first persistence/read-model layer for
explicitly designed active routes. It is not a transparent middleware that makes
every dynamic SQL query fast automatically.

BEST:

- Define the route that must be fast.
- Define the hot entity policy or projection contract for that route.
- Register generated JDBC-backed bindings when existing SQL rows must be warmed.
- Run warm/backfill before moving traffic.
- Keep archive/export/history paths as explicit SQL routes.
- Measure route latency, data parity, Redis memory, projection lag, and write-behind backlog before cutover.

ANTI-PATTERN:

- Mark every table as an entity.
- Expect Redis to load the whole database on startup.
- Let broad entity queries fall through to SQL and hydrate unbounded Redis data.
- Use full aggregates for list screens when a projection/read-model is required.

## Upgrade Notes From v0.1.0

- Update Maven dependencies to `0.2.0`.
- If a route depends on existing SQL rows, use `registerJdbcBacked(...)` and an explicit warm plan.
- If a route can read through on controlled misses, configure `ReadThroughConfig` with a strict `maxQueryLoadRows`.
- If using MSSQL, retest route-shaped warm queries because SQL Server rejects duplicate `ORDER BY` columns.
- If soft-delete columns are nullable, upgrade to this version before relying on write-behind upsert paths.
