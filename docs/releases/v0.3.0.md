# v0.3.0

Release tag: `v0.3.0`

## Release Focus

CacheDB `0.3.0` hardens the Redis-first runtime against stale hydration,
unbounded query materialization, write acknowledgement ambiguity, and
multi-pod cache-admission races. Existing public constructors and method
signatures remain available through delegating compatibility overloads.

## Added

- Version-aware JDBC source loading and atomic Redis compare-and-set hydration
  for warm, read-through, and external-change paths.
- Explicit stale-write and Redis-backpressure failure categories.
- JDBC read and write statement timeouts.
- Bounded admin request/background queues, request-body limits, and Redis-backed
  job-status visibility.
- Candidate-set limits for indexed and degraded query paths.
- Redis Function source hashing, semantic deployment versions, and a
  lease-protected deployment path.
- Focused Docker integration tests for stale hydration, tenant admission races,
  broad query rejection, function drift, and admin payload limits.

## Changed

- Durable compaction payloads, pending state, and version fences no longer use
  expiry. The durable compaction stream cannot be trimmed before acknowledgement.
- Prefix and token indexes are opt-in by default.
- Query paths reject broad candidate sets before materializing large JVM lists.
- Missing indexed payloads are repaired through a bounded loader or rejected;
  incomplete result pages are not returned silently.
- Tenant row and payload admission is serialized per tenant across pods.
- Repository graphs and CacheDB-owned Redis clients are reused and closed
  deterministically.
- Redis Function deployment rejects same-version source drift and version
  downgrades.

## Fixed

- An older warm, read-through, or CDC result cannot overwrite newer Redis state
  or resurrect a newer tombstone.
- A zero-row SQL write is no longer acknowledged without checking the persisted
  version.
- A tenant reservation evicted during concurrent admission cannot be re-added
  after the quota decision.
- Rejected admin jobs no longer remain indefinitely visible as queued.
- Production test runners use one explicit environment-resolution contract.

## Upgrade Notes From v0.2.0

1. Update all CacheDB modules and the annotation processor to `0.3.0` together.
2. Keep `redisFunctions.libraryVersion` aligned with the deployed library
   source. The `0.3.0` default must not be used with modified Lua source unless
   the deployment version is also changed.
3. Leave durable compaction payload, pending, and version TTL values at `0`.
   Non-zero values are rejected because expiry can violate replay safety.
4. Review routes that depended on prefix/token indexes. They are disabled by
   default and should be enabled only for bounded, measured routes.
5. Review broad entity queries. Candidate sets above
   `maxMaterializedCandidateIds` now fail fast instead of allocating an
   unbounded JVM list.
6. Custom source loaders that should hydrate Redis must expose positive entity
   versions through the versioned loader contract.
7. Set read-through and write-behind SQL timeouts according to the provider and
   route SLO; do not leave database calls unbounded.

## Recommended Dependency

```xml
<properties>
    <cachedb.version>0.3.0</cachedb.version>
</properties>

<dependency>
    <groupId>com.reactor.cachedb</groupId>
    <artifactId>cachedb-spring-boot-starter</artifactId>
    <version>${cachedb.version}</version>
</dependency>
```

MSSQL applications should also include `cachedb-storage-mssql` with the same
version.

## Validation Evidence

- Full Maven reactor: `222` tests discovered, `0` failures, `0` errors, and
  `10` environment-conditional skips on the local Docker validation run.
- PostgreSQL 18 and Redis 8.2.1 integration paths passed.
- Redis outage/recovery coordination passed before and after an injected
  container pause.
- Public API compatibility, Turkish documentation quality, release package,
  and sample consumer builds are release gates for this version.

These results are framework release evidence. Every consuming application must
still validate route parity, memory budget, provider latency, rollback, and the
target production HA topology with its own workload.
