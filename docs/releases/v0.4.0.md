# v0.4.0

Release tag: `v0.4.0`

CacheDB `0.4.0` makes the normal application path declarative without hiding
the production contracts. Consumers use one generated domain scope, configure
entity-specific active-data policies, and invoke typed query, relation,
projection, command, and warm operations without manual repository wiring.

## Highlights

- Package-level `GeneratedCacheModule.Scope` as the single typed domain API.
- Lazy, thread-safe entity scopes that avoid constructing unused repositories.
- YAML-backed `CachePolicyCatalog` with composite policies and fail-fast entity validation.
- Two-phase JDBC source and relation-loader registration that preserves per-entity policies.
- `ProjectionSchema` as one contract for projection serialization and indexed columns.
- Typed dry-run, projection-only, and full warm execution.
- Bounded incremental planner telemetry instead of startup-time full Redis scans.
- PostgreSQL and MSSQL samples with no manual repository or policy factories.
- Backward-compatible `metadata-only` starter default; JDBC loading remains an explicit choice.

## Validation

- Full 13-module reactor passed on Java 21.
- 83 integration tests passed against isolated Redis 8 and PostgreSQL.
- All 27 production benchmark, recovery, coordination, fault-injection, and certification tests passed.
- Both REST samples passed unit tests and live Redis 8 plus SQL-provider integration tests.

## Production Boundary

CacheDB still does not infer which routes should live in Redis. Each consumer
must define bounded route contracts, active-data policies, warm coverage,
memory budgets, SQL fallback paths, and rollback evidence before cutover.
